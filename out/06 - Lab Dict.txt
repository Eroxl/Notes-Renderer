10:T10e85,<p><mjx-container class="MathJax" jax="CHTML" style="font-size: 121%;" display="true"><mjx-math display="true" style="margin-left: 0; margin-right: 0;" class=" MJX-TEX"></mjx-math></mjx-container></p>
<p>This lab is all about making you think with dictionaries. Dictionaries (aka <em>maps</em> or <em>associative arrays</em>) are an abstract data type which stores pairs of data and can support operations like <code>insert</code>, <code>find</code>, and <code>remove</code>. We generally call these <em>(key, value)</em> pairs. The idea is to associate a <em>key</em> with a certain <em>value</em>. Hence, dictionaries are useful for when you want to quickly &quot;lookup&quot; a value associated with a certain key. The name <em>dictionary</em> comes from the more familiar physical dictionary, where you look up the definitions of words. If we were to model a physical dictionary with an ADT dictionary the keys would be the words, and the values would be their definitions.</p>
<p>Since dictionaries are abstract in nature, they have no prescription for what underlying structure should be used to implement them. Common choices are tree-based structures (e.g. balanced binary search trees, BTrees, etc.) or some sort of hash table. Different underlying implementations have different performance considerations, and which you use in the real world will depend on things like desired performance and what is readily available.</p>
<p>For this lab, we're not going to concern ourselves with the underlying implementation of dictionaries. Instead, we'll be using dictionary types that have already been defined for us. In C++ there are two &quot;built-in&quot; dictionary types: <a href="http://en.cppreference.com/w/cpp/container/map"><code>std::map</code></a> and <a href="http://en.cppreference.com/w/cpp/container/unordered_map"><code>std::unordered_map</code></a>. Both exist in the wonderful land of the Standard Template Library (STL). The STL is a subset of the C++ Standard Library (the library a compiler must implement to be standard-compliant) which provides data containers. As the name implies, these containers are templated and thus can contain any type (which meets the restriction of the container). You've probably experienced at least one of the STL types so far: <a href="http://en.cppreference.com/w/cpp/container/vector"><code>std::vector</code></a>, which is pretty much just a super fancy array. In general, the STL types are very convenient as they provide a lot of functionality (e.g. sorting, automatic resizing) that you don't have to bother implementing yourself.</p>
<p>Dictionaries are amongst some of the most important abstract data types that are used in the real world. Therefore, this lab is perhaps one of the most useful labs from a real world perspective. Mastering the concepts that you see in this lab will be a tremendous help for most technical interviews and programming problems in the future.</p>
<h1><code>auto</code> - Easy Types</h1>
<p>Most of the code we present in this class uses old (C++03) style, but in this lab we encourage you to use some C++11 features to make your life easier.</p>
<p>The first helpful C++11 tidbit we'll talk about is <code>auto</code>. STL functions are really big on iterators and templates. Unfortunately, these types can get quite verbose. For example, if I have an <code>std::unordered_map&lt;int, string&gt;</code> (which maps an integer to a string value) called <code>m</code> and I want to get the value associated with the key <code>5</code> (if it exists), I have to do something like this:</p>
<pre><code><span class="hljs-type">int</span> key \= <span class="hljs-number">5</span>;
unordered\_map&lt;<span class="hljs-type">int</span>, string\&gt; m;
unordered\_map&lt;<span class="hljs-type">int</span>, string\&gt;::iterator lookup;

lookup \= m.<span class="hljs-built_in">find</span>(key); /\* unordered\_map::find returns an iterator to a std::pair. \*/
<span class="hljs-keyword">if</span> (lookup != m.<span class="hljs-built_in">end</span>()) { /\* If the key isn<span class="hljs-string">&#x27;t in the map, m.end() is returned. \*/
    cout
        &lt;&lt; &quot;key: &quot;
        &lt;&lt; lookup\-&gt;first /\* the &#x27;</span>first<span class="hljs-string">&#x27; item in the pair is the key \*/
        &lt;&lt; &quot; value: &quot;
        &lt;&lt; lookup\-&gt;second /\* the &#x27;</span>second<span class="hljs-string">&#x27; item in the pair is the value \*/
        &lt;&lt; endl;
} else {
    cout &lt;&lt; &quot;key &quot; &lt;&lt; key &lt;&lt; &quot; not found&quot; &lt;&lt; endl;
}
</span></code></pre><p>Check out the type of <code>lookup</code>: <code>unordered_map&lt;int, string&gt;::iterator</code>. Yikes. The trouble with the STL is that every time we want an iterator we have to specify the full type of the container and then add <code>::iterator</code>. This can get pretty annoying. Thankfully, in C++11, there's <code>auto</code>. Instead of the above, I could do this:</p>
<pre><code><span class="hljs-type">int</span> key \= <span class="hljs-number">5</span>;
unordered\_map&lt;<span class="hljs-type">int</span>, string\&gt; m;

<span class="hljs-keyword">auto</span> lookup \= m.<span class="hljs-built_in">find</span>(key); /\* unordered\_map::find returns an iterator to a std::pair. \*/
<span class="hljs-keyword">if</span> (lookup != m.<span class="hljs-built_in">end</span>()) { /\* If the key isn<span class="hljs-string">&#x27;t in the map, m.end() is returned. \*/
    cout
        &lt;&lt; &quot;key: &quot;
        &lt;&lt; lookup\-&gt;first /\* the &#x27;</span>first<span class="hljs-string">&#x27; item in the pair is the key \*/
        &lt;&lt; &quot; value: &quot;
        &lt;&lt; lookup\-&gt;second /\* the &#x27;</span>second<span class="hljs-string">&#x27; item in the pair is the value \*/
        &lt;&lt; endl;
} else {
    cout &lt;&lt; &quot;key &quot; &lt;&lt; key &lt;&lt; &quot; not found&quot; &lt;&lt; endl;
}
</span></code></pre><p>Instead of specifying the type of <code>lookup</code>, we now specify it as <code>auto</code>. <code>auto</code> replaces the type information and does something called &quot;type inference&quot;. What this means is that the compiler will look at the type of the expression on the right hand side of the assignment and &quot;fill in&quot; the type information of the variable for you. Cool stuff, right? This means that for things like <code>find</code> we don't have to type out the nasty iterator type information. Now, don't go crazy with <code>auto</code>. A good rule of thumb is to only use it when it makes your code more readable. Since things like STL objects are pretty common it's often nice to use <code>auto</code> when dealing with functions that return iterators. For this lab I would type out the full type information the first few times (just so that you remember what the actual type is) and then after that use <code>auto</code>. Something that can be nice to do when using <code>auto</code> is to indicate (partially) what the type is in the name. Something like <code>lookup_itr</code> to indicate that this variable is really an iterator.</p>
<h1><code>map::operator[]</code> / <code>unordered_map::operator[]</code> - Easy Element Access</h1>
<p>A convenient way to access elements in a map is with <code>operator[]</code> (just like array subscripts). However, you have to be careful. If <code>key</code> doesn't exist in <code>m</code>, <code>m[key]</code> will create a default value for the key, insert it into <code>m</code> (and then return a reference to it). <strong>Because of this, you can't use <code>[]</code> on a <code>const map</code>.</strong></p>
<h1>Range-based Loops - Easy Iteration</h1>
<p>The next super cool thing in C++11 which is useful in this lab is the <a href="http://en.cppreference.com/w/cpp/language/range-for">range-based <code>for</code> loop</a>, also called for-each loops. As it turns out, a lot of programs involve iterating over collections of data (e.g. an array) and doing something with each of those values. The duty of a programming language is to make programming easier for us humans, so there's usually some sort of construct in the language to accomplish this. In C++03 the main construct was <code>for</code> loops with iterators. So if I had a <code>map&lt;int, string&gt; m;</code> and I wanted to iterate over every <code>(key, value)</code> pair I would have to do something like this:</p>
<pre><code>map&lt;<span class="hljs-type">int</span>, string\&gt; m;
<span class="hljs-keyword">for</span> (map&lt;<span class="hljs-type">int</span>, string\&gt;::iterator it \= m.<span class="hljs-built_in">begin</span>(); it != m.<span class="hljs-built_in">end</span>(); it++) {
    cout &lt;&lt; it\-&gt;first &lt;&lt; <span class="hljs-string">&quot;, &quot;</span> &lt;&lt; it\-&gt;second &lt;&lt; endl;
}
</code></pre><p>C++11 gives us a better way, taking advantage of what's called a &quot;for each loop&quot; or &quot;range-based for loop&quot;:</p>
<pre><code>map&lt;<span class="hljs-type">int</span>, string\&gt; m;
<span class="hljs-keyword">for</span> (std::pair&lt;<span class="hljs-type">const</span> <span class="hljs-type">int</span>, string\&gt; &amp; key\_val : m) {
    cout &lt;&lt; key\_val.first &lt;&lt; <span class="hljs-string">&quot;, &quot;</span> &lt;&lt; key\_val.second &lt;&lt; endl;
}
</code></pre><p>The way you should read that <code>for</code> loop is: &quot;for each <code>key_val</code> in <code>m</code> do ...&quot;. What's happening is that the <code>for</code> loop will update the variable <code>key_val</code> in each iteration of the loop to be the &quot;next&quot; element. Since we are iterating over a <code>map</code>, the elements are <code>(key, value)</code> pairs.</p>
<p>Notice that the type of <code>key_val</code> isn't an iterator: it's the &quot;dereferenced&quot; value of an iterator, in this case a reference to a <code>std::pair&lt;const int, string&gt;</code>. This value is a reference for two reasons: (1) so we don't make useless copies, and (2) so we can modify the values in the map. When we combine the ranged-for loop with <code>auto</code>, we get nice, readable code:</p>
<pre><code>map&lt;<span class="hljs-type">int</span>, string\&gt; m;
<span class="hljs-keyword">for</span> (<span class="hljs-keyword">auto</span> &amp; key\_val : m) {
    cout &lt;&lt; key\_val.first &lt;&lt; <span class="hljs-string">&quot;, &quot;</span> &lt;&lt; key\_val.second &lt;&lt; endl;
}
</code></pre><p>These kinds of for loops are simple, convenient, and intuitive to use. They have their limits, but for many iterating applications in C++ they're probably the best choice.</p>

Since we already know the value of <mjx-container class="MathJax" jax="CHTML" style="font-size: 121%;"><mjx-math class=" MJX-TEX"><mjx-mn class="mjx-n"><mjx-c class="mjx-c35"></mjx-c></mjx-mn><mjx-mo class="mjx-n"><mjx-c class="mjx-c21"></mjx-c></mjx-mo></mjx-math></mjx-container>, however, we can save quite a bit of computation (and time) by directly computing <mjx-container class="MathJax" jax="CHTML" style="font-size: 121%;"><mjx-math class=" MJX-TEX"><mjx-mn class="mjx-n"><mjx-c class="mjx-c36"></mjx-c></mjx-mn><mjx-mo class="mjx-n"><mjx-c class="mjx-c21"></mjx-c></mjx-mo><mjx-mo class="mjx-n" space="4"><mjx-c class="mjx-c3D"></mjx-c></mjx-mo><mjx-mn class="mjx-n" space="4"><mjx-c class="mjx-c36"></mjx-c></mjx-mn><mjx-mo class="mjx-n" space="3"><mjx-c class="mjx-cD7"></mjx-c></mjx-mo><mjx-mn class="mjx-n" space="3"><mjx-c class="mjx-c35"></mjx-c></mjx-mn><mjx-mo class="mjx-n"><mjx-c class="mjx-c21"></mjx-c></mjx-mo></mjx-math></mjx-container>, using the known value of <mjx-container class="MathJax" jax="CHTML" style="font-size: 121%;"><mjx-math class=" MJX-TEX"><mjx-mn class="mjx-n"><mjx-c class="mjx-c35"></mjx-c></mjx-mn><mjx-mo class="mjx-n"><mjx-c class="mjx-c21"></mjx-c></mjx-mo></mjx-math></mjx-container>.</p>
<p>This idea is essentially what memoization is.</p>
<blockquote>
<p>Memoization is an optimization technique used primarily to speed up computer programs by storing the results of expensive function calls and returning the cached result when the same inputs occur again.</p>
</blockquote>
<p>Now let's look at how we do this in code by looking at the <code>memoized_fac</code> function in <code>fac.cpp</code>. On line 94, we find:</p>
<pre><code><span class="hljs-type">static</span> map&lt;<span class="hljs-type">unsigned</span> <span class="hljs-type">long</span>, <span class="hljs-type">unsigned</span> <span class="hljs-type">long</span>\&gt; memo \= { {<span class="hljs-number">0</span>, <span class="hljs-number">1</span>} };
</code></pre><p>The left hand side is creating a variable named <code>memo</code> which is a <a href="http://en.cppreference.com/w/cpp/container/map"><code>std::map</code></a> that associates keys of type <code>unsigned long</code> to values of type <code>unsigned long</code>. In our case, we're going to map a key <mjx-container class="MathJax" jax="CHTML" style="font-size: 121%;"><mjx-math class=" MJX-TEX"><mjx-mi class="mjx-i"><mjx-c class="mjx-c1D45B TEX-I"></mjx-c></mjx-mi></mjx-math></mjx-container> to the value <mjx-container class="MathJax" jax="CHTML" style="font-size: 121%;"><mjx-math class=" MJX-TEX"><mjx-mi class="mjx-i"><mjx-c class="mjx-c1D45B TEX-I"></mjx-c></mjx-mi><mjx-mo class="mjx-n"><mjx-c class="mjx-c21"></mjx-c></mjx-mo></mjx-math></mjx-container>.</p>
<p>The <code>static</code> keyword is a neat little thing that basically makes this local variable remain initialized for the lifetime of the program so that we can reuse it across multiple calls to the function. It's like a global variable, but it's <em>only</em> accessible by that function.</p>
<p>The right hand side initializes the map to contain a key-value pair where the key is <code>0</code> and the value is <code>1</code>. Notice that this key-value pair corresponds to the base case of our mathematical definition of <mjx-container class="MathJax" jax="CHTML" style="font-size: 121%;"><mjx-math class=" MJX-TEX"><mjx-mi class="mjx-i"><mjx-c class="mjx-c1D45B TEX-I"></mjx-c></mjx-mi><mjx-mo class="mjx-n"><mjx-c class="mjx-c21"></mjx-c></mjx-mo></mjx-math></mjx-container> given above.</p>
<p>Alternatively, we could have used</p>
<pre><code><span class="hljs-type">static</span> map&lt;<span class="hljs-type">unsigned</span> <span class="hljs-type">long</span>, <span class="hljs-type">unsigned</span> <span class="hljs-type">long</span>\&gt; memo;
memo\[<span class="hljs-number">0</span>\] \= <span class="hljs-number">1</span>;
</code></pre><p>This disadvantage of this method is that now the key <code>0</code>'s value will be changed to the value <code>1</code> during every function call, as opposed to just at initialization as in the first example.</p>
<p>Line 97 is</p>
<pre><code><span class="hljs-keyword">auto</span> lookup \= memo.<span class="hljs-built_in">find</span>(n);
</code></pre><p><code>memo.find(n)</code> uses the <a href="http://en.cppreference.com/w/cpp/container/map/find"><code>std::map::find</code></a> function. As mentioned above, this method returns an iterator; the type of <code>lookup</code> is <code>std::map&lt;unsigned long, unsigned long&gt;::iterator</code>.</p>
<p>Lines 98–104 are:</p>
<pre><code><span class="hljs-keyword">if</span> (lookup != memo.<span class="hljs-built_in">end</span>()) {
    <span class="hljs-keyword">return</span> lookup\-&gt;second;
} <span class="hljs-keyword">else</span> {
    <span class="hljs-type">unsigned</span> <span class="hljs-type">long</span> result \= n \* memoized\_fac(n \- <span class="hljs-number">1</span>);
    memo\[n\] \= result;
    <span class="hljs-keyword">return</span> result;
}
</code></pre><p>Going back to what we already know about iterators and <code>std::map::find</code>, a return value of memo.end() basically implies that the key <code>n</code> does not exist in the map. In this case, we need to compute <code>n!</code> and update our map to store that value, which is exactly what we do in the <code>else</code> branch. A <code>std::map::iterator</code> is basically a pointer to a <code>std::pair&lt;const key_type, mapped_type&gt;</code> (<a href="http://en.cppreference.com/w/cpp/utility/pair"><code>std::pair</code></a>). In the <code>if</code> branch, <code>lookup-&gt;second</code> returns the value stored in the map for key <code>lookup-&gt;first</code>, which is going to be <code>n</code> in our case. If we wanted to be clever, we could rewrite this code as:</p>
<pre><code><span class="hljs-keyword">if</span> (lookup \== memo.<span class="hljs-built_in">end</span>()) {
    memo\[n\] \= n \* memoized\_fac(n \- <span class="hljs-number">1</span>);
}
<span class="hljs-keyword">return</span> memo\[n\];
</code></pre><p>Memoizing the factorial function doesn't actually do much for us in terms of computation time. To see this for yourself you can run:</p>
<pre>make fac
time ./fac 20
</pre><p>and compare it to the time it takes with:</p>
<pre>time ./fac 20 \-m
</pre><h1>Fibonacci</h1>

<p>This part of the lab involves making a dictionary for looking up anagrams of a given word. Two words are anagrams of one another if we can rearrange the letters of one to form the other. For example, the letters in the word &quot;dog&quot; can be rearranged to form the word &quot;god&quot;. We want an object (built from a word list) whom I can feed the string <code>&quot;dog&quot;</code> and it will spit out both <code>&quot;dog&quot;</code> and <code>&quot;god&quot;</code>.</p>
<p><strong>TASK 4:</strong> You'll find the definition of the <code>AnagramDict</code> class in <code>anagram_dict.h</code>. Implement the functions <code>add_word_to_dict</code> (which will complete the two provided constructors), <code>get_anagrams</code> and <code>get_all_anagrams</code>. <code>AnagramDict</code> has an instance of map named <code>dict</code> which maps a <code>string</code> to a <code>vector</code> of <code>string</code>s.</p>
<p>Remember that when using dictionaries we need to figure out a way to uniquely identify each of our values. In this case we want to uniquely identify a group of words which are anagrams of one another: we need some sort of function <code>f(key)</code> that we can apply to a key such that <code>f(&quot;dog&quot;) == f(&quot;god&quot;)</code> and <code>f(&quot;silent&quot;) == f(&quot;listen&quot;) == f(&quot;tensil&quot;)</code>, but <code>f(&quot;dog&quot;) != f(&quot;silent&quot;)</code> Phrased differently, you'll need to come up with some sort of function, such that it always returns the same value for all words that are anagrams of one another.</p>
